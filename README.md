# CATCH — Covert Asset Transfer Correlation Heuristics

CATCH is a pipeline for tracing Ethereum addresses across on-chain token transfers and detecting MEV activity (sandwich attacks, arbitrage) associated with those addresses.

---

## Quick start

```bash
bash run.sh
```

This runs the full pipeline: fetch raw block data → parse transfers → trace a target address with MEV detection.

Edit `run.sh` to change the target address, block range, or detection mode.

---

## Pipeline overview

```
fetch.py  →  raw/*.json
                ↓
           parse.py  →  parsed/*.csv  +  transfers.db
                              ↓
                         trace.py  →  stdout (cluster + MEV report)
```

1. **fetch.py** pulls raw block data from a JSON-RPC endpoint and writes one JSON file per block under `raw/`.
2. **parse.py** reads those JSON files, extracts ERC-20 Transfer events and Uniswap V2 Sync events, and writes per-block CSVs to `parsed/` and a consolidated SQLite database `transfers.db`.
3. **trace.py** loads `transfers.db`, BFS-traces a target address through all token transfers, and optionally detects sandwich attacks or arbitrage transactions along the way.

---

## Directory structure

```
CATCH/
├── fetch.py            — fetch raw block JSON via JSON-RPC
├── parse.py            — parse raw JSON → CSV + SQLite
├── trace.py            — BFS address tracer with MEV detection
├── run.sh              — end-to-end pipeline runner
├── transfers.db        — consolidated transfer database (generated)
├── raw/                — raw block JSON files (generated)
├── parsed/             — per-block CSV files (generated)
└── contracts/
    └── exchange.csv    — known DEX pool addresses and names
```

`raw/`, `parsed/`, and `transfers.db` are generated outputs and can be regenerated by running the pipeline.

## Scripts

### fetch.py

Fetches block data from a JSON-RPC node and saves it as `raw/<block_number>.json`.

```
python3 fetch.py <block> [--end <block>] [--url <rpc>] [--full-tx] [--receipts] [--logs] [--traces]
```

| Argument | Default | Description |
|---|---|---|
| `block` | — | Start block number |
| `--end` | same as start | End block (inclusive); fetches a range when set |
| `--url` | Arbitrum Sepolia | JSON-RPC endpoint URL |
| `--full-tx` | off | Include full transaction objects (default: hashes only) |
| `--receipts` | off | Also fetch transaction receipts |
| `--logs` | off | Also fetch event logs |
| `--traces` | off | Also fetch internal traces (requires debug/trace API) |

**Example — fetch blocks 100–105 with logs:**
```bash
python3 fetch.py 100 --end 105 --logs --full-tx
```

---

### parse.py

Parses raw JSON files from `raw/` and extracts token transfer rows (ERC-20 Transfer events + ETH transfers). Uniswap V2 Sync event data is attached to each row for reserve-based analysis.

```
python3 parse.py [<file> | --all] [--db [path]]
```

| Argument | Default | Description |
|---|---|---|
| `file` | — | Path to a single raw JSON file |
| `--all` | — | Process every JSON file in `raw/` |
| `--db [path]` | `transfers.db` | Also write rows to a SQLite database |

**Examples:**
```bash
# Parse a single block
python3 parse.py raw/100.json --db

# Parse all fetched blocks into the database
python3 parse.py --all --db
```

The database schema is:

```
transfers (
    token_address     TEXT,   -- ERC-20 contract address
    from_address      TEXT,
    to_address        TEXT,
    value             TEXT,   -- raw integer amount (no decimals applied)
    transaction_index INTEGER,
    transaction_hash  TEXT,
    block_number      INTEGER,
    reserve_data      TEXT    -- hex-encoded Uniswap V2 Sync reserves, if present
)
```

Re-running `parse.py` on the same block is safe; existing rows for that block are replaced.

### trace.py

Loads `transfers.db` and BFS-traces a target address outward through token transfers. Every reachable non-exchange address is added to the cluster. Exchange (DEX pool) addresses act as boundaries — they are recorded but not expanded — unless MEV detection is enabled and the swap exceeds the screening threshold.

```
python3 trace.py <address> [--db <path>] [--exchanges <path>] [--depth <n>] [--mode sandwich|arb]
```

| Argument | Default | Description |
|---|---|---|
| `address` | — | Target Ethereum address to trace |
| `--db` | `./transfers.db` | Path to the SQLite transfer database |
| `--exchanges` | `./contracts/exchange.csv` | Path to the known DEX pool CSV |
| `--depth` | 100 | Maximum BFS depth |
| `--mode` | off | MEV detection mode: `sandwich` or `arb` |

**Examples:**
```bash
# Plain BFS trace (no MEV detection)
python3 trace.py 0xABCD...

# Detect sandwich attacks
python3 trace.py 0xABCD... --mode sandwich

# Detect arbitrage
python3 trace.py 0xABCD... --mode arb
```

#### MEV detection modes

Both modes share the same **screening step**: when the tracer encounters an address swapping tokens through a known DEX pool, it computes

```
ratio = amount_in / (2 × r_in_before)
```

where `r_in_before` is the pre-swap input-token reserve, back-calculated from the post-swap Uniswap V2 Sync event. If `ratio > 0.01` (i.e. the swap is larger than 1% of the pool's one-sided liquidity), the transaction is flagged for further inspection.

| Mode | What it looks for |
|---|---|
| `sandwich` | A frontrun buy and matching backrun sell by the same address bracketing the flagged swap in the same block. The sandwicher's profit (backrun received − frontrun paid, in the input token) is reported. |
| `arb` | A multi-hop cyclic swap in a later transaction of the same block that passes through the same pool — a sign that the flagged swap created an arbitrageable price imbalance. |

When an attacker or arbitrageur is identified, their address is added to the cluster and traced in subsequent BFS iterations.

#### contracts/exchange.csv

A CSV with at least an `address` column listing known DEX pool addresses. An optional `name` column labels each pool. Addresses not in this file are treated as regular EOAs and expanded during BFS.

> **Important:** `exchange.csv` must be kept up to date for MEV detection to work correctly. The tracer identifies sandwich and arbitrage patterns by recognising pool addresses as DEX boundaries. Any pool address missing from this file will be treated as a regular wallet — the tracer will expand into it during BFS, the screening step will never fire for swaps through it, and sandwiches or arbitrages on that pool will go undetected. When analysing a new network or a new block range, make sure all relevant pool addresses are listed here first.

## run.sh

```bash
#!/bin/bash
mkdir -p contracts parsed raw
python3 fetch.py --full-tx --logs 2585770 --end 2585772
python3 parse.py --all --db
python3 trace.py 0xEA57D4a208e9F24150F404928357172Daf6fA60a --mode sandwich
```

Edit the block range, RPC URL, target address, and `--mode` flag here to rerun the pipeline on different data.
